# WARP — Workspace Autonomous Reservation Program

## Summary

WARP is an open-source **desk reservation system** (hot-desking / hotelling) built during COVID-19. It lets organizations manage office desks, parking stalls, and other bookable spaces through a visual, web-based interface. It's a monolithic Flask + PostgreSQL application with a vanilla JS frontend (bundled via Webpack), deployed via Docker with uWSGI and optionally nginx.

| iCal | Manual RFC 5545 generation (no library) |
| Auth | Built-in password hash (Werkzeug), plus LDAP3, MSAL (Azure AD), and Apache mod_auth_mellon (SAML2.0) |
| Reports | **xlsxwriter** for Excel export |
| Container | **Docker** with Alpine; uWSGI app server; optional nginx reverse proxy |

## Architecture

### Backend (Python/Flask)

The app is a **Flask application factory** (`warp/__init__.py`):

```
create_app()
  ├── initConfig()              # Load settings class + WARP_* env vars
  ├── db.init()                 # Bind Peewee tables to PostgreSQL; auto-initialize schema + run migrations
  ├── Register blueprints:
  │   ├── ical.bp               # /calendar/<login>/events.ics + action endpoints
  │   ├── view.bp               # / (HTML page routes)
  │   └── xhr.bp                # /xhr/* (JSON API)
  └── Register auth blueprint (exactly one):
      ├── auth.bp               # Built-in password login
      ├── auth_ldap.bp          # LDAP/AD login
      ├── auth_aad.bp           # Azure AD login
      └── auth_mellon.bp        # SAML2.0 (Apache mod_auth_mellon)
```

**Configuration**: `config.py` defines three settings classes (`DefaultSettings`, `DevelopmentSettings`, `ProductionSettings`). All config keys are overridable via `WARP_*` environment variables, parsed through `json.loads()` (supports arrays, dictionaries, booleans, null).

### Database Schema (PostgreSQL via Peewee)

The ORM uses Peewee's `Table` class (not models) for lightweight queries:

| Peewee `Table` | PostgreSQL table | Purpose |
|:---|:---|:---|
| `Blobs` | `blobs` | Binary data (zone map images) |
| `Users` | `users` | User accounts (login, password hash, name, role) |
| `Groups` | `groups` | Many-to-many user groups |
| `Zone` | `zone` | Bookable zones/floors/areas (name, type, map image ref) |
| `ZoneAssign` | `zone_assign` | User-to-zone role assignments |
| `Seat` | `seat` | Individual seats (name, X/Y coords on map, enabled) |
| `SeatAssign` | `seat_assign` | Seat-to-user assignment (with days_in_advance limit) |
| `Book` | `book` | Booking records (user, seat, from/to timestamps) |
| `UserPrefs` | `user_prefs` | Per-user settings (default zone, iCal token, reminders) |
| `UserToZoneRoles` | MATERIALIZED VIEW | Recursive expansion of zone roles through groups |
| `CalendarCache` | `calendar_cache` | Unlogged cache for generated iCal feeds |

**Key DB features**:
- A PL/pgSQL trigger (`book_overlap_insert`) prevents double-booking via an exclusion check on same-seat OR same-user within the same zone_group.
- `user_to_zone_roles` is a materialized view (WITH RECURSIVE) that expands group memberships. Triggers on `zone_assign` and `groups` refresh it.
- Schema migrations run automatically on startup (tracked in `db_initialized.version`). Currently at version 9.

**Account types**: `10=admin`, `20=user`, `90=blocked`, `100+=group`

**Zone types**: `10=DISABLED`, `20=ENABLED`, `30=PUBLIC_VIEW`, `40=PUBLIC_BOOK`

**Zone roles**: `10=ADMIN`, `20=USER`, `30=VIEWER`

### Frontend (JavaScript)

- **Build**: Webpack 5 with multiple entries (one per view page + a shared `base` bundle). Outputs to `warp/static/dist/` with content hashes. HTML includes are auto-generated into `warp/templates/headers/`.
- **Framework**: No SPA framework — traditional multi-page app with per-page scripts. Uses Materialize CSS for UI components.
- **i18n**: `node-polyglot` library; translations are JSON objects in `warp/static/i18n/` (5 languages).
- **Zone map**: Seats rendered as sprites on a background map image. Supports pan/zoom in the zone editor. Uses `noUiSlider` for time range selection.
- **Tables**: `tabulator-tables` 6.x for sortable, filterable data tables.

### Key Application Flows

1. **Booking a seat**: User navigates to a zone → sees seat map with existing bookings → selects date/time via slider and checkboxes → clicks a seat → `POST /xhr/zone/apply` with `book` action → DB validates overlap → iCal cache invalidated.

2. **Auto-book**: Floating "+" button → `POST /xhr/zone/autoBook/<zid>` → `runAutoBook()` finds the best available seat using tiered ranking (direct assignment > everyone > none), respecting `days_in_advance` windows and usage history.

3. **iCal integration**: `GET /calendar/<login>/events.ics?t=<token>` → returns RFC 5545 iCalendar feed with booking events + reminder events + one-click release/book URLs (HMAC-signed). Cached per-day in `calendar_cache` (UNLOGGED table).

4. **Admin workflows**: Zone management (add/edit/delete zones, upload map images, place seats), user management (add/edit/delete users, assign groups), group management, seat assignment (including "everyone" virtual access and per-assignment booking windows).

5. **Report**: Admin-accessible bookings list with filter-by-date, export to Excel (xlsxwriter).

## Deployment

**Preferred**: Docker Compose (`containers/compose/compose.yaml`) — 3 services: PostgreSQL, WARP (uWSGI), nginx.

**Docker build**: Multi-stage — first stage compiles JavaScript (Node.js) and builds Python wheel; second stage is Alpine runtime with only the wheel installed.

**uWSGI**: 4 processes, 2 threads, binding on port 8000.

**Without Docker**: Flask dev server with `flask --app warp --debug run` on port 5000.

## Noteworthy Design Decisions

1. **Single-user-per-zone-group constraint**: A user can only have one active booking at a time within a `zone_group` (enforced by `book_overlap_insert` trigger). This allows groups like "floors" and "parking" to coexist without shared booking rules.

2. **Materialized view for zone roles**: `user_to_zone_roles` recursively expands group memberships, giving efficient joins for permission checks. Refreshed by triggers on writes to `zone_assign` and `groups`.

3. **Time handling**: Timestamps are stored as wall-clock fake-UTC integers (`timegm(localtime_in_plan_tz())`). Each plan has an IANA `timezone` column (default UTC; set per-plan in the Plans admin). A `book_utc` SQL view converts wall-clock integers to real UTC instants for cross-TZ conflict detection. iCal feeds emit per-booking `TZID` stamps and RFC 5545 `VTIMEZONE` blocks; reminder events are gridded in each reminder zone's own plan TZ.

4. **Everyone seat assignment**: Seat assignments can have a `NULL` login (virtual "everyone"), meaning any user in the zone can book it. The sentinel key `__everyone__:550e8400-...` is used frontend-only.

5. **Auto-book tiered ranking**: Priority: (1) seats the user already booked on overlapping days in this zone, (2) directly assigned seats, (3) everyone/unassigned seats. Within each tier, sorted by user's past seat usage (least-used first). Respects `days_in_advance` booking windows.

6. **iCal security**: Action URLs (book/release) use HMAC-SHA256 with per-user ical_token. Nonces prevent replay. Confirmation is a two-step process (first shows confirmation page, second executes the action).

7. **No approval workflows** by design — the system is meant to be autonomous and management-free.

8. **Per-plan timezone support** — each plan has an IANA timezone; cross-plan conflict detection uses real UTC instants via the `book_utc` view.
